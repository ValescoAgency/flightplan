# Writing Agent Briefs

An **Agent Brief** is the comment posted on a Linear issue when it transitions to `ready-for-agent`. It is the **input** the AFK pre-flight reads to derive a goal-contract. The original issue body is context; the Agent Brief is the contract source-of-truth.

Pre-flight will reject a contract whose source brief is vague. So this format isn't decorative — every section maps to a goal-contract field that pre-flight checks.

## How the brief maps to the goal-contract

(Schema reference: `valesco-platform/afk/schemas/goal-contract.v1.json`. Fields below quote the schema.)

| Brief section            | Goal-contract field                          | Pre-flight rule                                       |
| ------------------------ | -------------------------------------------- | ----------------------------------------------------- |
| **Summary** + **Current/Desired behavior** | `intent.description`              | `minLength: 20`, durable, no file paths               |
| **Acceptance criteria**  | `intent.successCriteria` (array)             | `minItems: 1`, each `minLength: 5`, testable          |
| **Out of scope**         | `intent.nonGoals` (array)                    | each `minLength: 5`                                   |
| **Key interfaces** — types/interfaces | `intent.anchors.interfaces`         | Anchors must still resolve at pre-flight              |
| **Key interfaces** — config shapes | `intent.anchors.configShapes`         | Anchors must still resolve at pre-flight              |
| **External docs**        | `intent.anchors.docs[]`                      | Each gets a `versionHash`; pre-flight flags drift     |
| **Likely paths**         | (informs `scope.requiredPaths`)              | Pre-flight invalidates if any glob resolves to zero files |
| **Tier**                 | `metadata.tier`                              | Inherited from `.afk/config.yml`                      |
| **Customer** (Tier 1 only) | `metadata.customer`                        | Required when `tier: 1`                               |
| **Canary plan** (Tier 1 only) | `canaryPlan`                            | Required when `tier: 1`                               |
| **Termination conditions** | `terminationConditions`                    | `minItems: 1`                                         |

A **Human Brief** uses the same shape plus a *Why human, not agent* paragraph. Use it for `ready-for-human`.

## Principles

### Durability over precision

The issue may sit in `Todo` with `ready-for-agent` for days or weeks. The codebase will move. Write the brief so it stays useful through renames and refactors.

- **Do** describe interfaces, types, and behavioral contracts
- **Do** name specific types, function signatures, or config shapes
- **Don't** reference file paths — they go stale (and pre-flight's `intent.description` rejects them by convention)
- **Don't** reference line numbers
- **Don't** assume the current implementation structure will remain

### Behavioral, not procedural

Describe **what** the system should do, not **how** to implement it. The agent will explore fresh.

- **Good:** "The `SkillConfig` type should accept an optional `schedule` field of type `CronExpression`"
- **Bad:** "Open `src/types/skill.ts` and add a schedule field on line 42"
- **Good:** "When a user runs `/triage` with no arguments, they see a summary of issues needing attention"
- **Bad:** "Add a switch statement in the main handler function"

### Testable acceptance criteria

`intent.successCriteria` is what pre-flight scrutinises hardest. Each entry must be independently verifiable.

- **Good:** "`list_issues(team: 'VA', status: 'Triage', label: 'needs-info')` returns issues whose most-recent comment is from the reporter"
- **Bad:** "Triage should work correctly"

### Explicit non-goals

Pre-flight populates `intent.nonGoals` from the brief's *Out of scope*. Naming non-goals prevents the agent from gold-plating and informs adversarial review.

### Anchors that resolve

Pre-flight invalidates the contract if any anchor (interface, config, doc URL) disappears. Only list anchors that exist now and are likely to keep existing.

## Template — Agent Brief

```markdown
> *This was generated by AI during triage.*

## Agent Brief

**Category:** Bug | Feature | Improvement
**Project:** <Linear project name, if applicable>
**Repo:** <repo name, e.g. ffl-live-draft>
**Tier:** 1 | 2 | 3 (from `.afk/config.yml`)
**Customer:** <required when Tier 1, omit otherwise>

**Summary:** one sentence — what should happen

**Current behavior:**
What happens now. For Bugs, the broken behavior. For Feature/Improvement,
the status quo the change builds on. ≥ 20 chars; no file paths.

**Desired behavior:**
What should happen after the work is complete. Be specific about edge
cases and error conditions.

**Acceptance criteria:**
- [ ] Specific, testable criterion 1
- [ ] Specific, testable criterion 2
- [ ] Specific, testable criterion 3

**Out of scope (non-goals):**
- Thing that should NOT be changed in this issue
- Adjacent feature that might seem related but is separate

**Anchors:**
- Interfaces / types: `TypeName`, `OtherType`
- Config shapes: `OAuthConfig`, `SkillMetadata`
- External docs:
  - <URL> — short description (Obsidian PRD / Linear Doc / ADR)

**Likely paths the agent will need to touch:**
- `src/<area>/**` — where the relevant logic lives now
- `tests/<area>/**` — where new tests will go

(These are hints. The agent will refine `scope.requiredPaths` at pre-flight.)

**Termination conditions:**
- CI green
- Validator pass
- All new acceptance criteria covered by tests

**Sensitive-path flags:** auth | migrations | billing | none
(If any are flagged, pre-flight applies elevated delay per governance G1.)

**Canary plan** *(Tier 1 only — required by schema):*
- Metrics: <e.g. error_rate, p95_latency>
- Thresholds: <metric → number>
- Rollback trigger: <command or Vercel action>
- Window: <minutes>
```

## Template — Human Brief

Same as Agent Brief, plus this required section:

```markdown
**Why human, not agent:**
- [ ] Repo is `afkEligible: false` (e.g. `valesco-platform` per §14.3 — AFK can't modify its own policies)
- [ ] Requires production access AFK doesn't have (Supabase prod, Vercel prod, payments)
- [ ] Requires manual / hardware / third-party testing not available to AFK
- [ ] Cross-repo coordination spanning teams (VA / MREG / FMF)
- [ ] Design judgment that doesn't reduce to testable acceptance criteria
- [ ] Other: <plain-English reason>
```

If the only reason is "I want to do this myself," check the *Other* box and say so. Don't reach for a fake governance reason.

## Examples

### Good Agent Brief (Bug, Tier 2)

```markdown
> *This was generated by AI during triage.*

## Agent Brief

**Category:** Bug
**Project:** Fantasy Football Offseason League Manager
**Repo:** ffl-live-draft
**Tier:** 2

**Summary:** Skill description truncation drops mid-word, producing broken output.

**Current behavior:**
When a skill description exceeds 1024 characters, it is truncated at exactly
1024 characters regardless of word boundaries. This produces descriptions
that end mid-word (e.g. "Use when the user wants to confi").

**Desired behavior:**
Truncation breaks at the last word boundary before 1024 characters and
appends "…" to indicate truncation.

**Acceptance criteria:**
- [ ] Descriptions ≤ 1024 chars are unchanged
- [ ] Descriptions > 1024 chars are truncated at the last word boundary before 1024 chars
- [ ] Truncated descriptions end with "…"
- [ ] Total length including "…" never exceeds 1024 chars

**Out of scope (non-goals):**
- Changing the 1024-char limit itself
- Multi-line description support

**Anchors:**
- Interfaces / types: `SkillMetadata`
- Config shapes: none
- External docs: none

**Likely paths the agent will need to touch:**
- `src/skills/**` — frontmatter parsing and validation
- `tests/skills/**` — new boundary tests

**Termination conditions:**
- CI green
- Validator pass

**Sensitive-path flags:** none
```

### Good Agent Brief (Feature, Tier 1)

```markdown
> *This was generated by AI during triage.*

## Agent Brief

**Category:** Feature
**Project:** MREG Platform
**Repo:** millsreg-sanity
**Tier:** 1
**Customer:** Mills Real Estate Group

**Summary:** SLA engine should classify inbound leads as Hot / Warm / Cold and start the response clock.

**Current behavior:**
Leads land in the inbox and sit until an agent manually reviews them. There
is no automatic classification or SLA timer.

**Desired behavior:**
On lead intake, classify the lead and start the appropriate SLA timer.
Hot → same-day reminder until closed. Warm → 48h touchpoint timer. Cold →
24h response timer. Missed timers raise an internal SLA breach event.

**Acceptance criteria:**
- [ ] A new lead with classifier inputs `{intent: 'buy', timeline: '<30d'}` is classified Hot
- [ ] A Hot lead missed > 24h emits an `SlaBreached` domain event
- [ ] A Warm lead's 48h timer resets when a touchpoint is logged against the lead
- [ ] Classification is deterministic given the same inputs (covered by unit tests)
- [ ] The classifier accepts a `LeadInput` shape and returns a `Classification` enum

**Out of scope (non-goals):**
- The notification delivery layer (email/SMS) — emits events only
- Agent-facing UI for SLA status
- Cross-lead aggregation / dashboards

**Anchors:**
- Interfaces / types: `LeadInput`, `Classification`, `SlaBreached`
- Config shapes: `SlaConfig` (thresholds per classification)
- External docs:
  - https://linear.app/valescoagency/project/mreg-platform-c4d22c17ab5d — MREG Platform PRD

**Likely paths the agent will need to touch:**
- `apps/millsreg-sanity/src/sla/**`
- `apps/millsreg-sanity/src/leads/**`
- `tests/sla/**`

**Termination conditions:**
- CI green
- Validator pass
- All acceptance criteria covered by tests
- Classification benchmark suite passes (deterministic over fixture set)

**Sensitive-path flags:** none

**Canary plan:**
- Metrics: `sla_breach_rate`, `classifier_error_rate`, `p95_lead_intake_latency_ms`
- Thresholds: `sla_breach_rate < 0.05`, `classifier_error_rate < 0.01`, `p95_lead_intake_latency_ms < 250`
- Rollback trigger: `vercel rollback --yes` against the millsreg-sanity production deployment
- Window: 30 minutes
```

### Bad Agent Brief

```markdown
## Agent Brief

**Summary:** Fix the triage bug

**What to do:**
The triage thing is broken. Look at the main file and fix it. The function
around line 150 has the issue.

**Files to change:**
- src/triage/handler.ts (line 150)
- src/types.ts (line 42)
```

This is bad because:

- No AI disclaimer
- No category, repo, or tier — pre-flight can't establish the contract envelope
- No customer when one is likely required
- Vague description, no acceptance criteria — pre-flight rejects (`successCriteria minItems: 1`)
- Pinned to file paths and line numbers that go stale (pre-flight `intent.description` convention violated)
- No anchors — `intent.anchors.interfaces` empty
- No non-goals — adversarial review has nothing to bound the agent against
- No termination conditions — `terminationConditions minItems: 1` violated
- No sensitive-path declaration — pre-flight has to guess
