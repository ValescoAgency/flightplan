# Linear Triage — AFK Funnel

The purpose of this skill is to **prepare Linear issues for AFK runs**. Every triage decision answers one question:

> *Can this issue become a goal-contract that AFK pre-flight will accept?*

If yes → produce an **Agent Brief** and apply `ready-for-agent`. That's the goal.
If not yet → identify what's missing and either ask the reporter (`needs-info`) or park it (`Backlog`).
If never → eject it cleanly: `ready-for-human`, `Canceled`, or `Duplicate`.

The other outcomes are **filters** that remove issues which don't qualify. The Agent Brief is the payload.

All Linear operations go through `mcp__plugin_productivity_linear__*`. Never use `gh` for issue triage — issues live in Linear.

## AI disclaimer

Every comment created by this skill **must** start with this disclaimer on its own line, before any other content:

```
> *This was generated by AI during triage.*
```

## AFK eligibility — the first gate

Before recommending any state, read `.afk/config.yml` at the working repo root.

| Field             | Meaning for triage                                                                                                          |
| ----------------- | --------------------------------------------------------------------------------------------------------------------------- |
| `afkEligible: false` | The repo is a control plane. **`ready-for-agent` is unavailable.** All actionable issues route to `ready-for-human`.        |
| `afkEligible: true` + `projectTier: 1` | Production / client-critical. Briefs must enable a Tier-1 contract → require `customer` and a `canaryPlan`. |
| `afkEligible: true` + `projectTier: 2` | Internal tools / staging. Standard brief. No canary required.                                                |
| `afkEligible: true` + `projectTier: 3` | Prototype / side project. Standard brief, but note the `tierExpires` date.                                  |

**Hard rule:** `valesco-platform` is `afkEligible: false` per governance plan §14.3 (the AFK control plane cannot rewrite its own policies). Issues filed against it can move to `ready-for-human` but **never** to `ready-for-agent`. State this up front whenever a `valesco-platform` issue is on the table.

If `.afk/config.yml` is missing in an otherwise-active repo, treat that as the first thing to fix — surface it to the maintainer rather than guessing.

## State model

### Status (Linear-native workflow)

| Status        | Type        | Meaning                                                                |
| ------------- | ----------- | ---------------------------------------------------------------------- |
| `Triage`      | `triage`    | Incoming. Has not been classified or has open questions.               |
| `Backlog`     | `backlog`   | Ejected from active triage but not killed. Will revisit later.         |
| `Todo`        | `unstarted` | **AFK-ready or human-ready.** Has a Brief comment.                     |
| `In Progress` / `In Review` / `Reviewed` | `started` | Active work. Off-limits to this skill — read-only.       |
| `Done`        | `completed` | Closed normally.                                                       |
| `Canceled`    | `canceled`  | Won't fix. Permanent ejection.                                         |
| `Duplicate`   | `canceled`  | Closed because another issue covers it.                                |

### State labels (apply at most one)

| Label              | Meaning                                                                                                              |
| ------------------ | -------------------------------------------------------------------------------------------------------------------- |
| `needs-info`       | Specific, named information is missing — without it we cannot construct a goal-contract. Status stays `Triage`.      |
| `ready-for-agent`  | An Agent Brief comment exists. AFK pre-flight should accept it. Status moves to `Todo`. **The funnel destination.**  |
| `ready-for-human`  | Cannot become AFK-eligible (governance gate, prod access, judgment that doesn't reduce to acceptance criteria). Status moves to `Todo`. |

### Category labels (apply exactly one)

- `Bug` — something is broken
- `Feature` — new capability
- `Improvement` — enhancement to existing capability

### Invariants

- Every actively-triaged issue has **exactly one** category label.
- An issue has **at most one** state label. Conflicts are invalid — flag and confirm before doing anything else.
- `In Progress` / `In Review` / `Reviewed` are read-only. Don't touch.

## State machine

The funnel direction is **toward `ready-for-agent`**. Each transition either advances toward it, parks the issue, or ejects it.

| From                      | To                                       | Direction | Trigger                                  | Action                                                                                    |
| ------------------------- | ---------------------------------------- | --------- | ---------------------------------------- | ----------------------------------------------------------------------------------------- |
| New (`Triage`, no labels) | + category, stays `Triage`               | enter     | Skill on first look                      | Apply category, post recommendation comment.                                              |
| `Triage`                  | + `needs-info`                           | park      | Maintainer (skill)                       | Specific missing info named. Skill posts Triage Notes with what's blocking the brief.     |
| `Triage` / `needs-info`   | `Todo` + `ready-for-agent`               | **advance** | Maintainer (skill, after grilling)        | Skill writes Agent Brief, removes `needs-info` if present, applies label, moves status.   |
| `Triage` / `needs-info`   | `Todo` + `ready-for-human`               | eject     | Maintainer (skill)                       | Brief documents *why human, not agent*. Same shape as Agent Brief plus the reason.        |
| `Triage` / `needs-info`   | `Backlog`                                | park      | Maintainer (skill)                       | Acknowledged but deferred. Brief reasoning posted.                                        |
| `Triage` / `needs-info`   | `Canceled`                               | eject     | Maintainer (skill)                       | Polite explanation. For `Feature`/`Improvement`, write `.out-of-scope/<concept>.md`.      |
| `Triage` / `needs-info`   | `Duplicate`                              | eject     | Maintainer (skill)                       | Comment links the canonical issue.                                                        |
| `needs-info`              | back to `Triage` for re-evaluation       | advance   | Skill (detects new reporter activity)    | Reporter has commented since the last Triage Notes — re-evaluate.                         |

The maintainer can override directly via "Quick state override" below; the skill should still flag unusual transitions (e.g. `Triage` → `Done` skipping `Todo` is suspicious).

## Workflow: show what needs attention

Order the queue by **distance to AFK-ready**, not by age. The user's time is best spent on issues nearest the funnel exit.

Query Linear (scoped to the inferred team unless asked otherwise) and group results:

1. **Promotable** — `Triage` with category, body has clear current/desired behavior, only missing the Agent Brief itself. (Skill should offer to draft the brief inline.)
2. **`needs-info` with new activity** — reporter has commented since the most recent Triage Notes. Could now be promotable. (Compare comment timestamps via `list_comments`.)
3. **Stale `needs-info`** — `needs-info` for > 7 days with no reporter activity. Candidates for nudging or ejecting to `Canceled`.
4. **Raw `Triage`** — no category yet. Still needs first classification.

For each issue: identifier (`VA-87`), title, age, category if any, and a one-line summary of the body.

Show counts per bucket. Let the maintainer pick.

```
list_issues(team: <team>, status: "Triage", limit: 50)
list_issues(team: <team>, status: "Triage", label: "needs-info", limit: 50)
```

For bucket 2, fetch comments and compare timestamps.

## Workflow: triage a specific issue

### Step 1 — gather context

- `get_issue(<id>)` — body, labels, project, status, dates
- `list_comments(<id>)` — every comment with author + timestamp
- Parse any prior Triage Notes / Agent Brief comments authored by this skill (they begin with the AI disclaimer) to recover prior progress
- **Read `.afk/config.yml` in the working repo** — establish AFK eligibility and tier *before* recommending anything
- Read all `.out-of-scope/*.md` files and check for matching concepts
- For non-trivial cases, use `Agent` with `subagent_type=Explore` to find related code, types, tests, and recent history

### Step 2 — present a recommendation

Lead with the AFK question:

> **Can this become AFK-eligible? What's blocking it?**

Then state:

- **AFK eligibility** — `afkEligible: true/false`, tier (`1`/`2`/`3`), customer if Tier 1, sensitive-path flags (auth/migrations/billing). For `valesco-platform`, surface §14.3 explicitly.
- **Category** — `Bug` / `Feature` / `Improvement`, with reasoning
- **State recommendation** — `ready-for-agent`, `ready-for-human`, `needs-info`, `Backlog`, `Canceled`, or `Duplicate`, with reasoning
- **Distance to AFK-ready** — if recommending `needs-info`, name the *specific* facts the goal-contract needs that the issue currently lacks (success criteria, anchors, paths, tier inputs)
- **Out-of-scope match** — "This looks like `.out-of-scope/<name>.md` — we rejected this before because X. Same call?"
- **Codebase findings** — relevant interfaces, configs, tests, recent history

Then wait. The maintainer may agree, override, ask to flesh it out, or want to discuss.

### Step 3 — bug reproduction (bugs only)

If category is `Bug`, attempt reproduction *before* any /domain-model session:

- Read the reporter's reproduction steps
- Trace relevant code paths
- Try to reproduce: run tests, execute commands, narrow the failure
- **Reproduced** — capture the observation. This becomes the brief's *Current behavior* and shapes the success criteria.
- **Could not reproduce** — say so. May be environment-specific, already fixed, or misreported.
- **Insufficient detail to attempt** — strong signal for `needs-info`.

### Step 4 — /domain-model session (if needed)

If the issue cannot yet produce a complete brief, run a /domain-model session targeted at filling the goal-contract gaps:

- An `intent.description` (≥ 20 chars, durable, no file paths)
- At least one concrete `successCriteria` entry (≥ 5 chars, testable)
- Relevant `anchors.interfaces` and `anchors.configShapes`
- Likely `requiredPaths` globs (the agent will refine; the brief just needs to point at the right surface)
- Tier-1 only: customer name and a starter `canaryPlan` (metrics, thresholds, rollback trigger, window)

The session output feeds directly into the Agent Brief template.

### Step 5 — apply the outcome

Always: **comment first, then labels/status.** This preserves reasoning before the issue moves.

| Outcome             | Comment to post                                                                                | Labels                                                                | Status        |
| ------------------- | ---------------------------------------------------------------------------------------------- | --------------------------------------------------------------------- | ------------- |
| `ready-for-agent`   | **Agent Brief** (see [AGENT-BRIEF.md](AGENT-BRIEF.md))                                         | + category, + `ready-for-agent`, − `needs-info`                       | `Todo`        |
| `ready-for-human`   | **Human Brief** — Agent Brief shape + required *Why human, not agent* paragraph                | + category, + `ready-for-human`, − `needs-info`                       | `Todo`        |
| `needs-info`        | **Triage Notes** — what's established + the *specific* missing facts (see "Needs-info output") | + category, + `needs-info`                                            | `Triage`      |
| Won't fix (Bug)     | Polite explanation                                                                             | + `Bug`                                                               | `Canceled`    |
| Won't fix (Feature/Improvement) | Polite explanation **and** link to the `.out-of-scope/` file                       | + category                                                            | `Canceled`    |
| Duplicate           | Link to canonical issue                                                                        | + category                                                            | `Duplicate`   |
| Backlog             | One-paragraph note of why we're acknowledging but deferring                                    | + category                                                            | `Backlog`     |

For a `Feature`/`Improvement` going to `Canceled`, write or update `.out-of-scope/<concept>.md` *before* the comment, so the comment can include a real link.

Linear MCP calls (sketch):

```
save_comment(issue: <id>, body: "<comment>")
save_issue(issue: <id>, labelIds: [...], stateId: <statusId>)
```

Resolve label IDs via `list_issue_labels`, status IDs via `list_issue_statuses(team)`. Cache within the session.

## Workflow: quick state override

When the maintainer says "move VA-42 to ready-for-agent", trust them. Skip /domain-model.

Still confirm before writing:

- Labels to add/remove
- Status to apply
- Whether to post a comment, and what kind

For `ready-for-agent` without a /domain-model session, ask: "Draft a brief Agent Brief from existing context, or skip the comment?" Default to drafting one — pre-flight will reject a contract derived from a vague brief.

For `valesco-platform` issues, refuse `ready-for-agent` and explain (§14.3). Offer `ready-for-human`.

## Needs-info output

When a brief cannot yet be written, name the *specific* facts the goal-contract is missing. Vague "please provide more info" is useless; the questions must let the maintainer construct an Agent Brief once they're answered.

```markdown
> *This was generated by AI during triage.*

## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we need to make this AFK-ready (@<reporter displayName>):**

- A specific success criterion, e.g. "When X happens, the user should see Y" — what's the testable outcome?
- Which interface or config shape is involved? (e.g. `OAuthConfig`, `SkillMetadata`)
- Steps to reproduce on a fresh checkout (for Bugs)
- Customer name, if this affects a Tier-1 production property
```

Capture **everything** resolved during the /domain-model session in "established so far" — that work must not be lost.

## Resuming previous sessions

When triaging an issue with prior triage activity:

1. `list_comments(<id>)` and find prior comments authored by this skill (they begin with the AI disclaimer).
2. Parse what was already established.
3. Check whether the reporter has answered any outstanding questions.
4. Show the maintainer an updated picture: "Here's where we left off; here's what the reporter said since."
5. Continue the /domain-model session from where it stopped — never re-ask resolved questions.
